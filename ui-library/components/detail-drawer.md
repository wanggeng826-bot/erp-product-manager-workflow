# DetailDrawer

ERP 项目默认的「查看详情」容器——从列表里点行 → 抽屉滑出 → 看完 → 关掉 → 列表保持原貌。

## Ant Design 对应

`Drawer` + `Descriptions` + `Tabs` + `Timeline`。

## 为什么默认用抽屉而不是新页面

来自 `ui-interaction-spec §5.2 / §8.2 / §9.2`：

> 查阅统一优先抽屉。
> 抽屉关闭后，列表滚动位置、筛选条件、勾选状态应尽量保留。

少跳转 = 操作不打断 = 高频用户效率高。

只有以下情况才用独立详情页：

1. 详情内容超过 5 个 tab
2. 详情里有大量长流程任务（多步骤审批、复杂编辑）
3. 详情需要全屏沉浸式工作

## 关键尺寸

- 宽度：`67vw`（屏幕 2/3，由 `--layout-drawer-width` 控制）
- 高度：`100vh`
- 头部、Tabs、底部操作区固定；中间 `__body` 滚动

## 内部结构

```
.c-drawer
├── .c-drawer__head            ← 标题 + 关闭 ×
├── .c-drawer__meta            ← 状态 tag + 关键摘要（金额、时间、渠道）
├── .c-drawer__tabs            ← Tabs：基础信息 / 商品 / 物流 / 操作日志
├── .c-drawer__body            ← 滚动内容区
│   ├── .c-section-title         分组小标题
│   ├── .c-desc                  字段描述列表（4 列网格）
│   └── .c-timeline              操作日志/历史
└── .c-drawer__foot            ← 底部操作区，固定不滚动
```

## 底部操作区

约定从左到右：

1. **关闭** —— 总是有
2. 次要动作（复制、打印、下载）
3. **主操作** —— `btn--primary`，一个为限

如果详情里需要展示「危险动作」（删除、退款），用 Dropdown 收到一个「更多」按钮里——不要直接暴露在底部。

## 状态覆盖

| 状态 | 适用？ | 实现 |
|---|---|---|
| 默认 | ✅ | 见 BEGIN COMPONENT |
| 加载 | ✅ | `__body` 内部用骨架 |
| 空 | ⚠️ | 单个对象不会"空"，但子 tab（比如"物流"未发货时）可空 |
| 错误 | ✅ | `__body` 替换为 Result 风格的错误提示 |
| 成功反馈 | ⚠️ | 底部操作执行后用 Message 反馈 |
| 禁用 | ✅ | 个别操作按钮禁用 + tooltip 说原因 |
| 无权限 | ✅ | 抽屉本身不该打开；如果误开，整体内容替换为「您无权查看」 |

## 交互细节

- 点遮罩 / 按 ESC 都能关闭
- 抽屉打开后焦点应自动落到关闭按钮（暂未实现，下次完善）
- 列表里的勾选状态、滚动位置、筛选条件**不能因为抽屉关闭被重置**
- 抽屉里的"返回列表"链接不需要——关闭就是返回

## Tabs 设计

| Tab | 内容 | 何时显示 |
|---|---|---|
| 基础信息 | 描述列表（c-desc） | 必有 |
| 商品明细 | 内嵌小表 | 订单/采购/退款类对象有 |
| 物流 | Timeline 或表 | 有发货行为的对象 |
| 操作日志 | Timeline（固定字段） | **关键业务对象必有**，遵循 OperationLog 规范 |

避免做 5+ tab——会让用户找不到主信息。如果业务上确实需要，考虑拆抽屉或独立详情页。

## 与 EditDrawer 的关系

DetailDrawer 是**只读**的。如果需要编辑：

- 字段少 → 字段右侧加铅笔图标，点开 InlineEdit
- 字段多 → 关掉 DetailDrawer，打开 EditDrawer（共享同一对象 id）

**不要**在 DetailDrawer 里直接把字段做成可编辑——会让"查看"和"编辑"两种状态混淆。

## 反模式

❌ 抽屉里再嵌另一个抽屉（深层级让人迷路）
❌ 底部操作区放 5+ 个按钮
❌ 抽屉占满整个屏幕（那就是新页面，不如直接跳）
❌ 关闭抽屉时刷新列表（用户失去上下文）
❌ 操作日志字段不遵循固定模板（私自加列）
