# PageHeaderBar

页面标题区——长在 ErpShell 内容区的最上方。

## Ant Design 对应

`Typography.Title` + `Space` + `Button`。

不直接用 Ant Pro 的 `PageContainer`，因为它默认会带太多次级元素（标签、内容、扩展区），ERP 高频列表页用不到那么重。

## 强约束（来自 ui-interaction-spec §4.2 / §6.2）

> 一个页面只能有一个主任务；同一屏内只能有一个最高优先级主操作。

落到这个组件就是：

- **`.btn--primary` 在 actions 区只能出现一次**
- 其他都要用 `.btn`（次要）或 `.btn--text`
- 危险动作（删除、批量取消）放到行操作或抽屉里，不放到标题区
- **`PageHeaderBar / Actions` 只允许作为透明布局容器**：`fill=none`、`stroke=none`、`effect=none`、`padding=0`、`radius=0`
- **`PageHeaderBar / Actions` 必须 `Hug contents`**，不允许固定宽高后再塞空白

## 何时显示描述（`__desc`）

| 场景 | 显示描述？ |
|---|---|
| 数据列表（订单、商品、采购） | ❌ 用户已经知道在看什么，不需要 |
| 业务规则复杂的配置页（权限、计费规则） | ✅ 一句话提示后果或边界 |
| 新功能首次上线 | ✅ 可临时加引导文案，2 周后撤掉 |

描述**最多两行**，超过两行说明这部分应该独立成帮助入口（`FieldHelp` 组件）。

## 何时显示数字（`__title-extra`）

```html
<span class="c-page-header__title-extra">共 12,486 单</span>
```

适用于「这个页面承载多少对象」是核心信息时——例如订单列表、商品库。
**不要**显示「待我处理的有 X 条」之类业务衍生数字，那些请放到 PageHeader 下方的状态卡或筛选标签里。

## 与 BatchActionBar 的关系

当用户在表格里勾选若干行后，主操作位**不要被替换**。BatchActionBar 应该独立悬浮在表格上方而非取代 PageHeader。这样：

1. 主操作（如「新建订单」）始终可见，符合稳定一致原则
2. 批量操作的"上下文性"明显——一看就知道是针对勾选项的

## 状态覆盖

| 状态 | 适用？ | 实现 |
|---|---|---|
| 默认 | ✅ | 见 BEGIN COMPONENT |
| 加载 | ❌ | 标题区不需要加载——加载下沉到内容区 |
| 空 | ❌ | |
| 错误 | ❌ | |
| 禁用 / 无权限 | ✅ | 主操作 `aria-disabled="true"` + `title` 说明原因 |

## 反模式

1. 不要两个 primary 按钮并排
2. 不要超过 24px 的标题字号
3. 不要写大段"使用说明"——业务后台不是消费品，用户不会读
4. 不要给 actions 容器加白底、圆角、阴影，避免出现“按钮外再套一个小卡片”
5. 不要把 actions 容器设成固定宽高 + 左上对齐，会造成右上角留一块无意义空盒
