# QueryFilterBar

列表页 / 报表页的筛选区。

## Ant Design 对应

`Form` + `Input` + `Select` + `DatePicker` + `Button`。

## 关键决策（来自 ui-interaction-spec §10）

- 搜索框放筛选区**首位**
- 高频筛选**直接展示**（最多 4-6 个）
- 低频筛选折叠到「更多筛选」，避免首屏成为「表单墙」
- 查询/重置按钮放筛选区，**不与 PageHeader 的主操作混放**
- 用户提交查询后，已选筛选用 chip 在筛选区下方回显

## 高频 vs 低频的判定

我们当前项目的判定标准：

| 字段类型 | 一般归类 |
|---|---|
| 搜索关键词 | 高频 |
| 主状态字段（订单状态、库存状态） | 高频 |
| 业务渠道、平台 | 高频 |
| 主时间字段（下单时间、入库时间） | 高频 |
| SKU / 商品 ID | 中频，看业务 |
| 金额范围 / 数量范围 | 低频 |
| 标签 / 自定义字段 | 低频 |
| 创建人 / 操作人 | 低频 |

**经验法则**：如果业务方说「查 X 时一般会用这个字段」，就是高频；说「偶尔会用」就是低频。

## 已选筛选标签（chip）

提交查询后，**必须**在筛选区下方用 chip 显示当前生效的筛选。这样：

- 用户知道现在结果是基于什么过滤的
- 单个 chip 可以单点删除（× 按钮）
- 一键「全部清空」可以快速重置

如果 chip 数 > 6 还在叠加，要提示「筛选过多可能漏数据，建议拆分查询」。

## 网格布局

默认 4 列。每列字段宽度自适应。

- 单字段超过 1 列宽度（如「订单金额 0 - 1000 元」）→ 用 `grid-column: span 2`
- 字段名过长 → label 上方放，控件下方放（当前默认就是上下结构）
- 表单密度可微调：`gap: var(--space-4) var(--space-5)`

## 状态覆盖

| 状态 | 适用？ | 实现 |
|---|---|---|
| 默认 | ✅ | 见 BEGIN COMPONENT |
| 加载 | ⚠️ 偶尔 | 重型筛选项（如远端搜索 Select）单独加 Spin |
| 空 | ❌ | 筛选区本身没有「空」概念 |
| 错误 | ⚠️ | 单字段校验错误用 Ant Design `Form.Item` 的红色提示 |
| 成功反馈 | ❌ | 查询结果反馈在 DataTablePanel |
| 禁用 / 无权限 | ✅ | 某些筛选项依赖权限——可整个隐藏对应字段 |

## 与 DataTablePanel 的协作

筛选区**只产生筛选条件**，不要直接渲染结果。提交查询时：

1. 把筛选条件收集为对象（实现层）
2. 触发 DataTablePanel 的 `data-action="refetch"`
3. 表格区切换到 loading 态
4. 拿到结果后渲染

界面上的标志：筛选 chip 永远显示**已生效**的条件（不是表单里当前输入的草稿）。这意味着用户调整下拉框后必须按「查询」才会反映到 chip。

## 反模式

❌ 把「查询」按钮放到 PageHeader 区
❌ 高频字段超过 6 个不折叠
❌ chip 区缺失（用户找不到当前生效的过滤）
❌ 重置只清空表单不刷新结果（应同时重置表单 + 重新查询）
❌ 把日期筛选改造成多个独立年/月/日字段（用 RangePicker / DatePicker）
