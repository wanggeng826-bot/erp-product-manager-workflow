# ERP 参考模式补充规范

## 1. 文档定位

本文件用于沉淀从参考站点中抽取出的高价值 ERP 页面模式与公共控件规范，作为《UI 交互规范主文件》的补充。

适用场景：

- 需要设计 ERP 列表页、详情页、编辑流程时
- 需要补充公共控件方案时
- 需要在 PRD 中说明字段说明、操作日志、导入导出等复杂模式时

本文件不替代主规范；未说明部分仍以主规范为准。

## 2. 提炼来源

本文件基于以下参考内容整理：

- 参考站点：https://top-nova-39643531.figma.site/
- 重点页面：规范概览、交互原则、布局规范、CRUD 示例、公共控件

## 3. 参考站点的核心结论

参考站点整体呈现出三条非常适合当前项目的主线：

- 高效：减少操作步骤，提升数据处理效率
- 清晰：信息层级分明，避免视觉疲劳
- 一致：统一交互模式，降低学习成本

后续新页面设计时，应优先检查是否符合这三条主线。

## 4. 页面模式补充

### 4.1 列表页三段式结构

参考站点中，列表页使用稳定的三段式结构：

1. 标题区
2. 筛选器区
3. 表格区

补充规则：

- 列表页默认按此三段式结构输出原型。
- 新建按钮默认放在标题区右侧。
- 查询按钮放在筛选区末尾，不与新建按钮混放。
- 统计信息与分页放在表格区上下文中，不额外打断主结构。

### 4.2 列表到详情的闭环方式

参考站点中的 CRUD 示例明确采用“列表为主、抽屉闭环”的模式。

补充规则：

- 查看：点击行操作“查看”，打开只读抽屉
- 编辑：点击行操作“编辑”，打开可编辑抽屉
- 新建：点击“新增”，打开新建抽屉或弹窗
- 删除：点击“删除”，使用 `Popconfirm` 二次确认

除复杂流程外，不为常规 CRUD 单独新开页面。

### 4.3 字段说明模式

参考站点把“字段说明”作为 ERP 表格的重要组成部分。

补充规则：

- 简单字段说明：使用悬浮提示
- 中等复杂字段：使用问号图标点击查看说明
- 字段较多时：提供统一的“字段说明”入口

适用建议：

- 指标型字段优先加说明
- 缩写字段必须提供说明
- 业务概念易混淆字段必须提供说明

## 5. 反馈与状态模式补充

### 5.1 反馈类型矩阵

参考站点将反馈分为三类：

- `Message`：轻量反馈，约 3 秒自动关闭
- `Notification`：重要信息提醒，适合需要用户注意但不打断当前页面的结果
- `Modal`：阻断性反馈，适合不可逆或严重问题确认

补充规则：

- 保存成功、复制成功、轻删除成功：优先 `Message`
- 审批结果、后台任务完成、系统通知：优先 `Notification`
- 不可逆删除、严重错误、权限风险确认：优先 `Modal`

### 5.2 加载状态矩阵

参考站点对加载状态做了明确区分：

- `Skeleton`：首次加载，且用户可预知内容结构
- `Spin`：等待时间短、结构不稳定或局部刷新
- `Progress`：长时任务，需要反馈进度

补充规则：

- 列表首屏加载默认优先 `Skeleton`
- 查询刷新、按钮提交默认优先局部 `Spin`
- 文件上传、批量导入、报表生成默认优先 `Progress`

## 6. 布局尺寸补充

### 6.1 框架布局尺寸

参考站点给出了非常实用的后台默认尺寸：

- 侧边栏宽度：`200px` 展开，`64px` 收起
- 顶部栏高度：`64px`
- 面包屑高度：`48px`

补充规则：

- 新页面原型默认按此尺寸体系估算结构比例
- 若后续实际项目框架尺寸不同，可在主规范统一调整，但同一项目内必须保持一致

### 6.2 页面与卡片间距

参考站点给出的间距建议如下：

- 页面容器内边距：`24px`
- 紧凑布局内边距：`16px`
- 同级卡片间距：`16px`
- 不同模块卡片间距：`24px`

补充规则：

- 业务后台默认使用 `24px` 页面内边距
- 同一信息组内控制在 `16px` 节奏
- 模块与模块之间拉开到 `24px`

### 6.3 表单布局

参考站点采用了典型的 Ant Design 表单比例：

- `labelCol: { span: 6 }`
- `wrapperCol: { span: 18 }`
- 表单项间距：`24px`

补充规则：

- 标准后台表单默认按此比例出原型
- 若字段特别长或说明特别复杂，可按模块局部调整，但不应每页都变

### 6.4 响应式基线

参考站点建议以 `lg` 桌面端为设计基准，并将 `1200px` 级别作为主要参考宽度。

补充规则：

- 当前 ERP 的原型默认按桌面端优先设计
- 若要兼容窄屏，只做信息裁剪和布局压缩，不重做一套交互逻辑

## 7. 公共控件补充

### 7.1 店铺选择器

店铺选择器必须沿用当前 ERP 的「BI 驾驶舱 / 销售运营概况」控件样式。

固定结构：

- 平台：展示平台图标和平台名称
- 事业部
- 站点
- 店铺名

层级顺序固定为：`平台 -> 事业部 -> 站点 -> 店铺名`。

补充规则：

- 平台层必须展示平台图标与平台名称
- 使用树形结构，支持展开 / 收起
- 支持关键词搜索过滤店铺
- 支持多选，默认多选粒度到店铺
- 父级勾选应联动子级选择，半选态需清晰
- 每级缩进建议 `20px`
- 最大高度建议 `400px`，超出后滚动
- 选择结果需要能回显平台、站点、店铺范围，避免用户误判数据口径

适用场景：

- 店铺筛选
- 权限分配
- 数据报表范围选择

### 7.2 日期筛选器

参考站点中的日期筛选器由三部分组成：

- 粒度选择
- 时间类型选择
- 日期范围选择

补充规则：

- 默认提供按天、按周、按月等粒度
- 默认支持多种时间类型，如付款时间、创建时间、发货时间
- 默认提供快捷项，如今天、昨天、最近 7 天、最近 30 天、本月、上月

### 7.3 报表时间对比控件

设计报表时，如果涉及时间对比，必须先提供时间粒度选择：

- 按天
- 按周
- 按月

时间对比必须提供两种模式：

- 标准对比模式
- 自定义对比模式

标准对比模式：

- 系统基于当前统计周期自动计算对比周期
- 默认展示上一周期环比
- 默认展示同比周期
- 按天查看当天数据时，展示昨日环比和上月今日同比
- 按日选择整个 `3 月` 时，上月同比应对应整个 `2 月`，而不是某一天
- 选择连续日期范围时，对比周期应按完整范围计算，保证本期、环比、同比的时间跨度一致

自定义对比模式：

- 用户可自定义勾选或选择对比时间
- 系统按用户选择的时间进行数据统计和对比
- 自定义对比时，需要明确展示本期时间与对比时间，避免混淆统计口径

控件设计规则：

- 不同时间粒度必须使用对应的时间控件样式
- 按天优先使用日期范围选择
- 按周优先使用周维度选择
- 按月优先使用月份范围选择
- 时间类型选择、时间粒度选择、日期范围选择、对比模式切换应位于同一筛选区域
- 标准对比 / 自定义对比建议使用分段式按钮或 `Radio.Group`，当前模式必须高亮
- 对比模式旁应提供帮助说明，解释环比、同比和自定义对比口径

### 7.4 报表指标卡片对比明细

如果报表设计指标卡片，并且卡片展示本期值、环比、同比等指标，必须支持鼠标悬浮展示当前时间和对比时间。

标准对比模式下，悬浮明细至少展示：

- 本期：时间范围、指标值、差值占位
- 环比：时间范围、指标值、差值和趋势
- 上月同比：时间范围、指标值、差值和趋势

自定义对比模式下，悬浮明细至少展示：

- 本期：时间范围、指标值
- 自定义对比：时间范围、指标值、差值和趋势

展示规则：

- 卡片本体优先展示核心指标值和关键变化率
- 详细时间范围和差值放入悬浮层，避免卡片过载
- 悬浮层需使用稳定表格化布局，便于横向比较
- 趋势颜色必须清晰区分上升、下降和持平
- 对于负向指标，颜色含义需要按业务语义处理，不能机械套用上涨为好、下降为坏
- 指标口径、时间范围、对比方式必须能映射回筛选条件

适用场景：

- 报表
- 订单查询
- 审批记录
- 库存分析

### 7.5 操作日志

参考站点中的操作日志设计非常适合作为 ERP 标准能力。

当前项目的操作日志必须严格基于用户提供的模板设计。

强制规则：

- 只能基于模板做删减，不能新增字段
- 不能新增模板外的记录字段
- 不能把模板中的字段改名为其他含义
- 若具体业务不需要某字段，可删减或留空，但不得扩展新列

固定模板字段如下。

操作列表字段：

- 菜单
- 一级功能
- 二级功能

记录字段：

- 操作人
- 操作类型
- 操作对象
- 操作详情
- 操作时间

说明：

- “菜单”用于标识业务模块入口，如学院管理、题库管理
- “一级功能”用于标识直接功能动作，如添加学院、编辑学院、进入题库
- “二级功能”仅在一级功能下还需再细分时使用
- “操作对象”用于记录业务对象名称与标识，如学院名称 `(id)`、题型 + 题目 id
- “操作详情”仅在确有必要时记录详细变化内容
- “操作时间”属于保留字段，默认建议记录

补充规则：

- 操作详情优先记录字段变化，不记录无意义描述
- 操作时间使用统一服务器时间
- 检索与筛选能力也必须基于上述模板字段建立，不能反向新增日志字段
- 敏感信息必须脱敏
- 普通用户与管理员的日志查看范围可不同

推荐文案格式：

- 修改：`字段名：将[旧值]修改为[新值]`
- 新增：可不写详情，必要时仅记录关键对象信息
- 删除：可不写详情，必要时仅记录关键对象信息
- 批量：可不写详情，必要时仅记录处理结果摘要

示例约束：

- 编辑学院时，可在“操作详情”中记录学院名称、管理员账号等字段变化
- 添加学院、停用学院、启用学院、删除学院时，可不写“操作详情”
- 添加题目、编辑题目、删除题目时，“操作对象”可写为 `题型 + 题目 id`
- 登录场景下，“操作对象”和“操作详情”可为空或使用 `—`

### 7.4 文件导入方案

参考站点中的导入流程采用三阶段结构：

1. 下载模板
2. 上传文件
3. 结果反馈

补充规则：

- 所有导入都应优先使用统一流程
- 上传文件后必须先做预检；存在强校验错误时，不得创建导入任务，不得显示可点击的“开始导入”，应提供预检报告下载和重新上传入口
- 上传后统一进入任务中心追踪处理结果
- 结果文件优先回写在原始数据旁边，而不是只给一句“导入失败”
- 结果下载建议增加两列：
  - 是否导入成功
  - 错误原因

导入页面默认要表达的能力：

- 模板下载
- 模板版本号
- 填写说明
- 文件约束
- 校验规则
- 结果下载

## 8. 在 PRD 中如何引用本文件

当需求涉及以下内容时，PRD 应明确引用本文件：

- 列表页结构
- 抽屉式查看/编辑
- 字段说明
- 日期筛选器
- 店铺选择器
- 操作日志
- 文件导入

推荐写法：

“本功能中的列表结构、字段说明方式、抽屉交互、导入反馈与操作日志展示，除业务特例外，均遵循《ERP 参考模式补充规范》。”
