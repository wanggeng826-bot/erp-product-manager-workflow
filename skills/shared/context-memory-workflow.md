# Context Memory Workflow

本文件把 `planning-with-files` 的长上下文方法翻译成当前产品经理工作流可执行的规则。

## 1. 文件分工

| 文件或目录 | 用途 | 何时更新 |
|------------|------|----------|
| `task_plan.md` | 根目录唯一活跃重任务的目标、阶段、状态、关键决策 | 仅在重任务启动后使用；阶段完成后；范围变化时 |
| `findings.md` | 根目录唯一活跃重任务的调研发现、证据、假设、页面来源映射、问题清单 | 每完成 2 次重要读取、搜索、浏览、截图或评审后 |
| `progress.md` | 根目录唯一活跃重任务做了什么、改了什么、验证了什么、遇到什么错误 | 每个阶段结束；重要文件修改后；验证后 |
| `knowledge/` | 长期稳定的用户偏好、公司背景、自研 ERP 规则、术语 | 任务结束时，从临时文件中蒸馏已确认且可复用的信息 |
| `cases/<case-name>/` | 某个具体业务案例的 PRD、原型、评审材料和交付物 | 处理具体案例时 |

核心边界：

- `task_plan.md`、`findings.md`、`progress.md` 只服务一个当前活跃的重任务，不是正式交付物，也不是跨任务总账。
- `knowledge/` 只存放稳定、已确认、跨任务复用的信息。
- 单个案例的业务细节、PRD、原型、截图和评审记录放在 `cases/<case-name>/`。
- 已完成任务的 planning 文件要归档到 `cases/<case-name>/planning/` 或 `cases/_workflow-history/<date>-<slug>/`，然后重置根目录 planning 文件。
- 不把单次任务里的猜测直接写成长期事实；不确定内容必须标记为 `待确认`。
- 新任务默认不读取历史 `cases/**`；只有用户明确选择继续或参考某个案例时，才读取该案例内容。

## 2. 启动规则

遇到以下任务时，先使用 planning files：

- 新需求、方案设计、PRD、竞品分析、原型生成。
- UI 评审、Ant Design 合规检查、正式原型优化。
- 需要 5 次以上工具调用、跨多个文件或跨多轮对话的任务。
- 用户要求“长上下文”“别丢上下文”“继续上次任务”“整理工作流”。

以下情况不要启用 planning files：

- 已有原型、小文案、小布局、小交互的局部调整。
- 单文件或单目录的窄范围修订。
- 只需要查看当前 diff、目标文件和直接关联 spec 就能完成的任务。

启动时至少写清：

- 目标：本轮最终要交付什么。
- 当前阶段：正在调研、设计、实现、验证还是交付。
- 范围边界：主线工作流、具体案例、知识沉淀分别放哪里。
- 案例边界：本次是新案例、历史案例迭代，还是只使用主线模板。
- 成功标准：什么检查通过才算完成。

启动前先做一次隔离检查：

- 如果根目录 planning 文件仍是上一个已完成任务，先归档，再重置。
- 只有确认是“重任务”后，才创建 `.codex/planning-active`。
- 轻任务不创建 `.codex/planning-active`，也不更新根目录 planning 文件。

新需求启动门禁：

- 第一轮只能做资料盘点、背景对齐、假设标注、信息缺口和确认清单。
- 用户确认前，不得输出方案草案、PRD 草案、原型方案或创建业务文档。
- 如果用户给了截图、竞品参考或零散描述，先反问并建立一致性锚点，不把零散材料直接扩写成方案。
- 只有用户明确说“进入方案”“生成 PRD”“创建文档”“继续输出”时，才进入下一阶段。

## 3. 产品经理任务的记忆循环

每轮复杂 PM 任务按这个循环执行：

1. 读 `task_plan.md`，确认目标、阶段和剩余工作；若文件与当前任务无关，先停止并归档。
2. 读相关 Skill 和必要 reference，不全量加载无关材料。
3. 每完成 2 次重要信息获取，就把关键事实写入 `findings.md`。
4. 做重要决策前，回看 `task_plan.md` 和 `findings.md`。
5. 阶段完成后，更新 `task_plan.md` 状态和 `progress.md`。
6. 任务结束时，只把稳定且已确认的规则蒸馏到 `knowledge/`，并把根目录 planning 文件归档后重置。

## 4. PRD 和原型专用记录

做 PRD 或原型前，`findings.md` 必须记录页面来源映射：

| 项目 | 记录要求 |
|------|----------|
| 页面数量 | 每个页面来自 PRD 哪个章节或用户哪条确认 |
| 页面关系 | 列表、详情、抽屉、弹窗、批量操作之间的关系 |
| 主任务 | 每个页面只保留一个最高优先级任务 |
| 可见元素 | 导航、页签、摘要卡、区块标题、按钮都要能回指来源 |
| 状态覆盖 | Loading、空、异常、成功、禁用、无权限等如何体现在真实 UI |
| 权限差异 | 通过隐藏、禁用、只读、提示等正式 UI 表达，不用角色切换器 |

禁止把以下内容作为正式页面的一部分：

- 演示控制区。
- 角色切换器。
- 页面状态切换器。
- 调试面板。
- 无 PRD 或用户确认来源的导航、页签、摘要卡、快捷操作、模块标题。

## 5. 错误和失败处理

任何失败都要写入 `progress.md` 或 `task_plan.md` 的错误区：

- 失败动作。
- 错误现象。
- 已尝试的修复。
- 下一步换用的方案。

同一个失败不能原样重复三次。第三次仍失败时，整理已尝试方案和阻塞点，向用户确认。

## 6. 交付前检查

交付前至少确认：

- `task_plan.md` 的阶段状态已更新。
- `findings.md` 有关键证据、来源映射和重要决策。
- `progress.md` 有修改记录和验证结果。
- 正式 PRD、原型或评审结论没有依赖未确认假设。
- 可复用的长期规则已经进入 `knowledge/`，单次任务过程没有进入长期知识库。
