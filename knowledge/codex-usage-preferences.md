# Codex Usage Preferences

## 已确认

- 用户是 Codex 重度使用者，但不希望自己承担 Git 分支、提交、回滚和上下文治理的复杂度。
- 用户担心 Codex 帮忙改了大量文件后，后续一旦出错无法追溯到原因、任务、文件和时间点。
- Codex 应主动承担守护职责：开始前检查分支和未提交状态，改动前说明文件范围，完成后给修改摘要、验证结果和 checkpoint 建议。
- 用户不希望每次都主动点名 `$codex-workflow-guardian`；Codex 应根据风险信号自行发现并启用。
- 用户希望 Codex 在每次执行或对话开始前预判风险；如果存在高风险配置或流程问题，应先暂停对话/执行，要求用户修正配置或确认更安全路径。
- 用户希望在使用过程中被温和提醒风险点，逐步形成更专业的 Codex 使用习惯。

## 长期执行偏好

- 重要任务默认执行可溯源闭环：`git status` -> 范围说明 -> 任务记录 -> 文件修改 -> 验证 -> diff 摘要 -> checkpoint 建议。
- 当任务涉及分支、未提交文件、跨目录修改、长对话继续、上下文过大、任务目标切换时，Codex 默认自动进入 guardian 模式。
- guardian 模式不仅提醒，还要有 Stop Gate：遇到错误分支、未 checkpoint 的大批改动、危险 Git 操作、敏感信息、缺失来源映射或关键配置不明时，先停止执行。
- 遇到长对话、多目标混杂、上下文过大、反复提醒同一约束时，Codex 应建议生成 handoff 并开启新对话。
- 具体业务案例不要污染主线工作流；案例资产放入 `cases/<case-name>/`。
- 主线能力建设放入 `skills/`、`knowledge/`、`ui-library/`、`docs/` 等通用位置。
- 对 Git/分支/提交解释要用低门槛语言，重点讲“怎么防丢、怎么回滚、怎么定位问题”，不要假设用户熟悉 Git 术语。

## 待持续提醒的风险点

- 长期在一个对话框内处理多个不同目标，会增加上下文膨胀和漏约束风险。
- 长期不提交 checkpoint，会导致后续无法快速定位是哪一轮改坏了。
- 把工作流规则、具体案例、临时实验混在同一批改动里，会增加回滚和复用成本。
- 原型没有 `prototype-spec.md` 和验证记录时，后续难以解释页面元素来源。
