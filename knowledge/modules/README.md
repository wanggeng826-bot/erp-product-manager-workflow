# 业务模块知识库

> 这里沉淀的是**关于我们这个 ERP 系统的稳定业务规则**——不是某次需求的产物，而是所有需求的副产品。

## 这个目录在解决什么

之前的问题：Claude 每次接到新需求都得问一遍"你们订单状态有几种"、"你们包裹拦截规则是什么"——重复劳动，用户烦。

这里的目标：每次方案讨论结束后，从中**提取关于系统本身的事实**，沉淀到对应模块文件里。3 个月后，新人或者新对话从零开始读这个目录，就能理解 80% 的业务。

## 何时新建一个模块

启发式：当一个领域**反复**出现在不同需求里、并且有自己独立的状态机/数据模型时，就给它单独开一个 `modules/<模块名>.md`。

不确定时，先放到 `business-rules.md` 里，等积累到 3-5 条再升级成独立模块。

## 单个模块文件的结构

每个模块文件按这个骨架写——空字段也要保留，标 `<待沉淀>`：

```
# [模块名]

## 1. 这个模块管什么 / 不管什么

## 2. 核心实体（字段表）

## 3. 关键状态机（mermaid）

## 4. 业务规则（每条独立成段 + 来源标注）

## 5. 与其他模块的关系

## 6. 常见误解 / 易混淆点

## 7. 历史决策（链接到 decisions/）
```

## 沉淀的三条规则

### 规则 1：每条事实独立成段 + 标注来源

❌ 不要：
> 订单状态有 5 种。订单超过 30 分钟未支付自动取消。订单号是 O- 开头的……

✅ 要：
> - 订单状态有 5 种：待支付、已支付、已发货、已完成、已取消。（来自 2026-01 收银台改版）
> - 订单超过 30 分钟未支付，系统自动取消并释放库存。（来自 2026-01 收银台改版）

**为什么**：以后某条事实要改，能精准定位；新人能从来源追溯当时的语境。

### 规则 2：冲突要标注，不要默默覆盖

新需求方案和已有知识库冲突时，**先在 PRD 里指出冲突**，让用户确认是：
- A) 旧的写错了 → 改 knowledge
- B) 这次要变更旧规则 → 保留旧值（划线）+ 新值 + 决策链接
- C) 两个共存（按条件区分） → 拆子规则

### 规则 3：先碎后整

不要追求一次写得完美。开始时就是一堆零散事实点。每过 5-10 个需求做一次整理：合并重复 / 归档失效 / 规范段落。

## 何时往这里写

| 触发时机 | 谁来写 |
|---|---|
| PRD 写完 + 用户确认 | Claude 主动从 PRD 里抽事实 |
| 原型反复改完，业务规则有更新 | Claude 在用户验收后主动同步 |
| 用户主动说"这条业务规则记到知识库" | Claude 当场记 |

> 一句话：**每次需求结束前，留一次工具调用沉淀知识库**。不沉淀 = 这次会话白做。
