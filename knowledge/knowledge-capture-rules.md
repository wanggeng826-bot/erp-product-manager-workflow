# Knowledge Capture Rules

## 什么应该进入知识库

- 用户多次表达或明确确认的偏好。
- 会长期影响 PRD、竞品分析、原型和 UI 判断的规则。
- 公司、业务、ERP、角色、流程、数据口径等稳定背景。
- 已确认的设计原则、交互规则、术语定义。
- 反复出现的评审标准和常见问题。

## 什么不应该进入知识库

- 单次任务的中间过程。
- 临时实验、未采用方案、短期 TODO。
- 未确认的猜测。
- 只属于某个案例的具体 PRD 或原型细节。
- 只属于某个历史案例的业务字段、页面版本、原型代码和评审材料。
- 敏感信息、账号、密钥、隐私数据。

## 如何记录不确定内容

- 不确定但有价值的信息写入 `待确认`。
- 使用时必须提醒用户它尚未确认。
- 一旦用户确认，再移动到 `已确认`。

## 知识更新节奏

- 当前任务过程写入 `task_plan.md`、`findings.md`、`progress.md`。
- 任务结束时，从短期文件中挑选稳定信息，蒸馏进 `knowledge/`。
- 每次重大工作流调整后，检查是否需要更新知识库索引。

## 使用优先级

1. 用户当前明确指令。
2. 当前 PRD 或任务文件。
3. `knowledge/` 长期知识库。
4. Skill 规则和 references。
5. 通用经验。

如果上述信息冲突，以更具体、更新、用户明确确认的信息为准。
