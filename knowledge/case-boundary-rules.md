# Case Boundary Rules

## 核心原则

主线工作流只保存通用规则、Skill、知识库和模板。具体业务需求、历史 PRD、原型版本、评审材料必须放到 `cases/<case-name>/`。

## 默认检索规则

- 新需求、竞品分析、方案输出、PRD 输出、原型生成默认不检索 `cases/**`。
- 只有用户明确表示“继续某个案例”“基于历史版本”“复用已有退款案例”时，才读取对应 `cases/<case-name>/`。
- 如果用户只是提出同类业务主题，例如“退款方案”，不能自动读取历史 `cases/refund-management/`。
- 如果需要判断是否已有案例，只能先列出案例名称并向用户说明存在历史案例，不能直接把历史 PRD 当成当前输入。

## 新案例规则

- 新业务需求必须先确认是新案例还是历史案例迭代。
- 新案例使用独立目录：`cases/<case-name>/`。
- 新案例的 PRD、截图、原型、评审材料都只能放在该案例目录内。
- 主线 `intake/prd/` 和 `prototype/` 只用于临时入口或通用演示，不承载长期案例资产。

## 历史案例规则

- 历史案例只能作为用户确认后的参考材料。
- 使用历史案例时，必须在 `findings.md` 记录“读取了哪个案例、为什么读取、哪些内容被复用”。
- 历史 PRD 不得覆盖当前用户新需求。
- 如果历史案例和当前需求冲突，以当前用户明确指令为准。

## 污染判定

以下情况视为污染：

- 具体业务 PRD 出现在主线 `intake/prd/`。
- 具体业务原型长期保存在主线 `prototype/`。
- Skill 或 `knowledge/` 中写入单个案例的业务细节。
- 新任务默认读取了无关历史案例并把历史内容当作当前需求。

以下情况不视为污染：

- 历史案例完整保存在 `cases/<case-name>/`。
- 主线文档只出现通用的案例管理规则。
- `knowledge/` 只沉淀跨案例稳定规则，不沉淀单个案例细节。
